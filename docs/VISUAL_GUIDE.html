What is Dremio Load?

A self-contained batch data ingestion tool with a full web UI

Dremio Load reads data from databases, cloud storage, SaaS tools, and ad platforms — then writes it into Dremio as Apache Iceberg tables. Everything runs in a single Docker container with a browser-based interface at http://localhost:7071.

🔌

25+ Connectors

Postgres, MySQL, S3, Snowflake, Google Ads, Salesforce, MongoDB, and more.

⚡

Incremental Load

Cursor-based incremental pulls. Only new or changed rows on each run.

🎯

Two Write Modes

Mode A: Dremio SQL MERGE. Mode B: PyIceberg direct write for high throughput.

📊

Pipeline Visibility

Visual source → job → target flow with run history, success rates, and row counts.

🔐

Secure by Default

Env vars, HashiCorp Vault, and masked credentials throughout the UI.

🤖

AI Agent

Optional Claude-powered agent for managing pipelines with natural language.

Installation

One Docker command — no Python or Node.js required on your machine

🐳
Docker Hub: mshainman/dremio-load:latest — available at hub.docker.com/r/mshainman/dremio-load

  1. Pull the image:
    docker pull mshainman/dremio-load:latest

  2. Create a persistent volume for jobs and config:
    docker volume create dremio-load-data

  3. Seed the required config file:
    docker run --rm -v dremio-load-data:/data busybox sh -c "echo 'jobs: []\ntarget: {}' > /data/config.yml"

  4. Start the container:
    docker run -d --name dremio-load -p 7071:7071 -v dremio-load-data:/data mshainman/dremio-load:latest

  5. Open http://localhost:7071 in your browser.

💡
Jobs persist across restarts. All jobs, run history, and offsets are stored in the dremio-load-data volume. Upgrade by pulling the new image and recreating the container — your data stays intact.

Configure Target (Dremio)

Set up where data will be written before creating any jobs

Click Target in the sidebar. There are two write modes — pick the one that matches your Dremio setup.

Mode A — Dremio SQL (recommended). Uses Dremio's REST API with SQL MERGE/INSERT statements. Works with all Dremio deployments including Dremio Cloud.
Mode B — PyIceberg Direct. Writes directly to an Iceberg REST catalog, bypassing Dremio SQL. Faster for large loads. Requires an Iceberg catalog URL and warehouse path.

  1. Enter your Dremio Host and Port (default: 9047).

  2. Enter Username / Password — or a Personal Access Token (PAT) for Dremio Cloud.

  3. Set the Catalog / Source and Schema where tables will be created.

  4. Choose Mode A (Dremio SQL) or Mode B (PyIceberg Direct).

  5. Click Test Connection, then Save.

⚠️
If Dremio is running in another Docker container, use the container name or Docker network IP — not localhost — as the host. Containers can't reach each other via localhost.

Create a Load Job

Define a source, its tables, and where to write the data

Click Jobs / Sources in the sidebar, then click + New Job in the top right.

Jobs / Sources page. Each card shows the job name, status badge, load mode, source type, last run time, and row count. The action buttons on the right let you run ▶, pause ⏸, edit ✏, delete 🗑, and expand the table list ∨.
New Load Job modal — Amazon S3 example. Fields change dynamically based on the selected source type. S3 shows bucket, prefix, region, and access credentials. File format auto-detects from the file extension.
Bottom of the job modal. Post-run hooks let you call a webhook on success or failure. The Transform Studio integration triggers a linked pipeline automatically after a successful load.

Key fields

FieldDescription
Job NameFriendly name shown on the job card and in run history
Source TypeThe connector to use — see the full list in the next section
Load ModeIncremental: only new/changed rows · Full: replace all data each run
ScheduleCron expression (e.g. 0 6 * * * for daily at 6am), or leave blank for manual-only
Target TableThe Dremio/Iceberg table to write to (e.g. "Catalog"."schema"."table"). Use Browse to pick from the catalog.
Tables / CollectionsComma-separated list of tables, views, or file prefixes to load

Google Ads — OAuth Setup

One-click OAuth flow — no manual token generation required

Google Ads requires OAuth 2.0 credentials. Dremio Load includes a built-in Connect with Google popup flow that handles the OAuth handshake and automatically fills in the refresh token — no command-line tools or copy-pasting needed.

Google Ads job form. Fill in your Developer Token, OAuth Client ID, Client Secret, and Customer ID — then click Connect with Google. The link "How to get Google Ads API credentials →" opens the Google documentation.
After authorizing in the popup. The form shows a green "Connected to Google Ads" badge with your Google account email. The refresh token is stored automatically. Click Re-authorize to switch accounts.

Step-by-step

  1. In Google Cloud Console, create an OAuth 2.0 Client ID of type Desktop app. Copy the Client ID and Client Secret.

  2. Get a Developer Token from Google Ads → Tools → API Center. Apply for Basic Access for production use (Test tokens only work against test accounts).

  3. In Dremio Load, create a new job with source type Google Ads. Fill in the Developer Token, Client ID, Client Secret, and Customer ID (digits only, no dashes).

  4. Click Connect with Google. A browser popup opens — sign in with your Google account and grant access.

  5. The popup closes automatically and the form shows the green Connected badge. Click Create Job.

Google Cloud Console. Create your project here and enable the Google Ads API. The Project ID and number are shown on the welcome screen.
OAuth 2.0 credentials page. Copy the Client ID and Client Secret from here into the Dremio Load job form. Choose Desktop app as the application type.
💡
Manager (MCC) accounts: Dremio Load automatically detects if your Customer ID is a manager account. It finds sub-accounts and routes metric queries accordingly. If no sub-accounts are linked yet, metric tables return 0 rows — this is expected behavior, not an error.

Available Google Ads tables

campaigns ad_groups ad_group_ads campaign_performance ad_group_performance search_terms keywords audience_performance geographic_performance

Running Jobs

Trigger manually or let the scheduler handle it

Job card after a successful run. Shows the green ✓ success badge, load mode (incremental), source type, last run timestamp, and row count. Action buttons: ▶ Run · ⏸ Pause · ✏ Edit · 🗑 Delete · ∨ Expand tables.

Action buttons explained

ButtonAction
▶ Play (green)Trigger the job immediately, regardless of schedule
⏸ PauseToggle enable/disable. Disabled jobs are skipped by the scheduler but can still be run manually
✏ PencilOpen the job form to edit any setting — name, connection, tables, schedule, target
🗑 TrashDelete the job (requires confirmation). Data already in Dremio is not affected
∨ ChevronExpand the card to see the full list of tables configured for this job
ℹ️
Incremental mode: After the first run, only rows where the cursor column (e.g. updated_at) is newer than the last run value are fetched. The cursor is stored in SQLite automatically. To force a full reload, use the reset icon in the Runs page.

Scheduler

See all jobs and trigger them on demand

Scheduler page. Jobs are grouped into Scheduled Jobs (with a cron expression) and Manual Jobs. Each row shows the job, source type, load mode, last run time, status, and a Run button for immediate execution.

Cron schedule examples

ExpressionMeaning
0 * * * *Every hour at :00
0 6 * * *Daily at 6:00 AM
0 */4 * * *Every 4 hours

Pipeline View

End-to-end visualization: source → job → target

Click Pipeline in the sidebar for a visual overview of every configured pipeline. Each card shows the full flow for one job — click any card to drill into the detail view.

Pipeline Overview. Each pipeline card shows three connected blocks: Source (type + table count) → Job (name, mode, schedule) → Target · Dremio SQL (host + tables). The status badge and success rate appear on the right. Click the source block to edit the job, the target block to configure the target, or anywhere on the card to open the pipeline detail page.
Multiple pipelines at a glance. Error state is shown in red with the error message inline below the card. Success shows 100% success rate and run count. You can see at a glance which pipelines are healthy and which need attention.

Pipeline Detail Page

Clicking a pipeline card opens the full detail view for that job, showing:

  • Source panel — connection summary, per-table row counts and last-run status icons
  • Job panel — mode/schedule chips, success rate bar, run history dots (hover for details), last run summary
  • Target panel — Dremio host, destination table, live Preview Target Data button
  • Per-table run history — the last 30 individual table loads with rows, duration, and error messages

Explorer

Browse your Dremio catalog and source data without leaving the UI

Explorer page — two tabs. Dremio Catalog browses the Dremio namespace tree and previews target tables. Source Data (shown) lets you select a configured source job, browse its tables, and preview raw source records before or after a load.

14

Copy Into

Dremio-native file ingestion — let Dremio do the reading

Copy Into page. Specify a target Iceberg table, the source location in a Dremio-registered source (e.g. @my_s3_source/folder/), file format (Parquet, JSON, CSV), and optional filename pattern regex. Click Preview SQL to see the generated statement before running.

Use COPY INTO when your files are already registered in Dremio as a source (S3, GCS, ADLS). Dremio reads the files natively — no Python parsing involved. Dremio tracks which files have already been loaded, so re-running is safe.

Settings

Notifications, secrets management, and AI Agent configuration

Settings → Notifications. Configure email alerts via SMTP and/or a Slack incoming webhook. Alerts are sent when a job fails. Click Send Test Alert to verify your configuration before saving.
Settings → Vault Secrets. Connect to HashiCorp Vault (token or AppRole auth). Once configured, use vault:path#field syntax in any connection credential. Environment variables use ${VAR_NAME} syntax and need no additional setup.
Settings → AI Agent. Enable the Claude-powered AI Agent by entering your Anthropic API key and selecting the model. When enabled, an AI Agent tab appears in the sidebar, letting users manage pipelines using natural language chat.
🔐
Secrets tip: Never put passwords directly in config files. Use ${ENV_VAR} for environment variables passed into the container, or vault:secret/path#key for Vault-managed secrets. Both are resolved at runtime and never stored in plain text.
